5/20/2016

Today, I coded the movement pattern for the balloons. To my surprise, this took the majority of the day, but at least I am very happy with the end result. I posted this code in the same GitHub repository that generates this website. Not that much time was spent writing it, but a lot of time was spent debugging it as well as plugging in magic numbers for all of my declared floats in the console that made me happy with the balloon's movement. The end result is a balloon that floats around in a sinusoidal pattern towards the player's base that smoothly transitions into a descent at a specified distance. Finally, the balloon creates an explosion when it hits the base. Part of the reason that I am posting this is because this code is the longest that I've directly written rather than modified for my needs (it's also one of the longest in general), since what I wanted from this code was fairly unique to my game. However, I have also spent a bit of time today thinking about my presentation, which made me realize that I should consider adding a piece of code to GitHub. As I plan on using this computer to present, I might just pull up MonoDevelop and show some piece of code there, since I am more interested in showing the Unity interface than I've been working with than explaining any code I write. Still, it could be nice to have here. The game itself is coming along pretty nicely. All I need to do is create spawners that generate these balloons and create a user interface that shows base damage in order for my game to feel like it has a functional purpose to it outside of being a fun flight simulator. I will probably turn towards the presentation soon rather than pushing out additional features that I want, as I can keep working on this project after the presentation.

5/18/2016

I managed to fix some more issues today. The main problem I had at the end of yesterday was the the explosion wasn't functioning correctly. After looking in the unity code directory some more, I discovered that Instantiate uses the origin rather that the transform of the object with the script attached as a default location for spawning objects. This caused my explosions to always appear off the map, and was not too hard to fix. Another problem that I had was that the bullets would sometimes fly through the terrain. There's no real way to stop this other than to slow down the speed a lot, but I did it as much as I could today while still looking good. I also looked at the directory a bit more in order to figure out how to make the audio work with 3D sound (which uses distance), but I am still figuring out the correct settings.

5/13/2016

It feels odd to transition from being completely guided through the process of making a game to suddenly being taken off of the leash. That said, I am at least making progress for now. I searched through the packages I found yesterday and this morning and began to figure out what I wanted to use in a game. What stood out to me by far the most was the airplane, which already had motion pre-programmed into it and had a lot of different components and animations to work off of. Making a flying game also seems like a great use of the unity space, since it makes good use of the Y axis, so that part is settled. I also discovered a type of asset called a terrain while searching through some of the assets today. As the plane covers ground very quickly, these terrains will be very important, as they are by far the most efficient way to create ground in Unity, as they are simply made out of a tile that gets repeated over and over again and a terrain collider. To make the scene look less repetitive, there is also a very efficient way to add trees, which were included in the standard assets. Terrain can also be raised, lowered, and retiled by using different sized brushes. I'm definitely very glad that I found these, and spent a significant part of the day playing around with them. Next week, I am probably going to have to get back into scripting. While the plane moves properly and responds to certain controls, I don't like the way that some of these controls are configured. I also learned how to change these due to a tangent in the Tanks tutorial. I've done some tweaking today, but not enough.

5/2/2016

I feel like I've learned a lot more today than I initially expected to. After reviewing some interface tutorials, I managed to complete the Rollong a Ball project. located at https://unity3d.com/learn/tutorials/projects/roll-ball-tutorial Although the product produced was very simple, actually running through the motions of creating a game gave me a much better idea of how many features of the program work, including rigid bodies which allow for physics simulations, materials that allow for coloring or texturing objects, and prefabs, which allow for the generation of identical objects within a game. I also got my first chance to script in C#, and founnd that my extremely basic knowledge of the syntax was sufficient for following the tutorial. Additionally, there is an extensive document with all of the unity specific functions, and hitting control + ' allows for research in a way very similar to writing a ? and then hitting tab in SageMath. After completing the project, I looked into some of the tutorials explaiining the concets of the physics engine, mostly as a review. There were a few options not covered in the Roll a Ball project though, such as the ability to prevent rotation in certain axes. Overall, Unity is more physics-based than I anticipated, but there will probably still be a good amount of time spent figuring out the coding. The project definitely feels much more doable after dippping my feet in and actually creating something with Unity.
